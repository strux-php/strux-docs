# Queue System

## Overview

Strux includes a robust, database-driven queue system designed for processing background jobs outside of the HTTP request lifecycle.

Queues allow you to offload long-running or non-blocking tasks such as:

- Sending emails
- Processing uploads
- Generating reports
- Syncing with external APIs
- Data imports and exports
- Webhook processing

By delegating these tasks to background workers, your application remains fast and responsive for end users.

---

## Why Use Queues?

Without queues:

- The user waits for the task to complete.
- Long processes block HTTP responses.
- Slow external services degrade performance.

With queues:

- Tasks are executed asynchronously.
- HTTP responses return immediately.
- Background workers handle processing safely and reliably.

:::info
Queues improve perceived performance and system scalability.
:::

# Getting Started

## Initialize the Queue Tables

Before using the queue, you must create the required database tables.

From your project root:

```bash
php bin/console queue:init
````

This command creates:

| Table         | Purpose                                |
| ------------- | -------------------------------------- |
| `jobs`        | Stores active and pending jobs         |
| `failed_jobs` | Stores jobs that exceeded retry limits |

---

# Configuration

Queue configuration is defined in:

```
src/Config/Queue.php
```

This configuration determines:

* Default queue connection
* Available drivers
* Retry behavior
* Failed job storage

---

## Default Connection

You can switch between:

| Driver     | Behavior                                        |
| ---------- | ----------------------------------------------- |
| `sync`     | Executes immediately (no background processing) |
| `database` | Uses background workers                         |

Example:

```php
'default' => env('QUEUE_CONNECTION', 'database'),
```

---

## Retry Behavior

The `retry_after` option defines how long (in seconds) a worker will wait before considering a job stalled.

Example:

```php
'retry_after' => 90
```

If a worker crashes or stalls:

* The job is automatically released
* It becomes available again
* Another worker may process it

# Queued Event Listeners (Recommended)

The most seamless way to use queues in Strux is through the Event System.

If an Event Listener implements:

```
ShouldQueue
```

Strux automatically dispatches it to the queue instead of executing it synchronously.

---

## How It Works

1. An event is dispatched.
2. Strux detects the `ShouldQueue` contract.
3. A background job is created.
4. The HTTP request continues immediately.
5. A worker processes the job asynchronously.

# Manual Jobs

If your task is not tied to the Event System, you can create and dispatch jobs manually.

A job is a simple class containing a `handle()` method.

---

## Dispatching a Job

You push a job onto the queue using the `Queue` service.

After dispatching:

* The job is serialized
* Inserted into the `jobs` table
* Awaiting processing by a worker

---

# Running the Worker

To process queued jobs:

```bash
php bin/console queue:work
```

The worker:

* Polls for available jobs
* Locks them safely
* Executes them
* Handles failures and retries

---

:::info Production Tip
The worker runs indefinitely.

In production, use a process monitor such as:

* Supervisor
* systemd
* PM2
* Docker restart policies

This ensures the worker restarts automatically if it crashes.
:::

---

# Atomic Locks & Concurrency

Strux’s database queue uses atomic row locking:

```
FOR UPDATE SKIP LOCKED
```

This guarantees:

* Multiple workers can run simultaneously
* No job is processed twice
* Safe concurrent processing

You can safely scale horizontally by running multiple worker processes.

---

# Job Lifecycle

Understanding the lifecycle helps with debugging and reliability.

### 1. Dispatch

The job object is serialized and inserted into the `jobs` table.

---

### 2. Polling

The worker searches for jobs where:

```
reserved_at IS NULL
```

---

### 3. Locking

The job row is locked to prevent duplicate execution.

---

### 4. Execution

* The job is unserialized
* Dependencies are resolved via the container
* The `handle()` method runs

---

### 5. Success

The job is permanently removed from the `jobs` table.

---

### 6. Failure & Retries

If an exception occurs:

* The job is released back into the queue
* The `attempts` counter increments
* The job is retried

After 3 failed attempts:

* It is removed from `jobs`
* Logged into `failed_jobs`
* The full exception stack trace is stored

---

# Failure Handling

Failed jobs are not lost.

You can:

* Inspect failure details
* Retry failed jobs manually

* Create monitoring dashboards

# Best Practices

## 1. Keep Payloads Small

Pass lightweight data such as:

* IDs
* Scalars
* Small arrays

Avoid:

* Full database models
* Large objects
* Heavy in-memory structures

Fetch fresh data inside `handle()`.

---

## 2. Design for Idempotency

Jobs may run more than once due to:

* Worker crashes
* Timeouts
* Retry logic

Ensure your job can safely execute multiple times without corrupting data.

---

## 3. Use Dependency Injection Properly

Constructor arguments are serialized into the database.

Dependencies in `handle()` are resolved fresh by the worker.

Preferred pattern:

* Constructor → minimal data
* `handle()` → injected services

---

## 4. Avoid Long Blocking Calls

If interacting with:

* External APIs
* Slow services
* Heavy computation

Implement:

* Timeouts
* Retry logic
* Graceful failure handling

---

## 5. Monitor Queue Health

In production, monitor:

* Job throughput
* Failure rate
* Worker uptime
* Table growth

Queues are infrastructure — treat them as critical systems.
