# Transactions: Keeping Your Data Consistent

A transaction is a group of database operations that must all succeed together. If any single operation fails, the entire group is rolled back — as if nothing ever happened.

Think of transactions like sending money between bank accounts:
1. Withdraw $100 from Account A
2. Deposit $100 into Account B

If step 1 succeeds but step 2 fails, you must undo step 1. A transaction handles this automatically.

> [!INFO]
> Transactions ensure **ACID** compliance: Atomicity, Consistency, Isolation, Durability.

---

## 1. Basic Transactions

Transactions are managed through static methods on the `Model` class. Any model class can start, commit, or roll back a transaction.

### Manual Transaction

```php
Model::beginTransaction();

try {
    // Subtract from sender
    $sender->balance -= 100;
    $sender->save();

    // Add to receiver
    $receiver->balance += 100;
    $receiver->save();

    // If we get here, both operations succeeded
    Model::commit();
} catch (\Throwable $e) {
    // If anything went wrong, undo everything
    Model::rollBack();
    throw $e;
}
```

### The `transaction()` Helper

The `Model::transaction()` helper handles begin, commit, and rollback automatically:

```php
Model::transaction(function () use ($sender, $receiver) {
    $sender->balance -= 100;
    $sender->save();

    $receiver->balance += 100;
    $receiver->save();
});
```

If the callable throws an exception, the transaction is rolled back automatically.

> [!TIP]
> You can call `Model::transaction()` on any model class — `User::transaction(...)`, `Product::transaction(...)`, etc. They all share the same underlying transaction because the nesting level is tracked statically.

---

## 2. Transaction Methods

| Method | Description |
|--------|-------------|
| `Model::beginTransaction()` | Start a new transaction (or create a savepoint if nested) |
| `Model::commit()` | Save all changes made during the transaction (or release savepoint) |
| `Model::rollBack()` | Undo all changes (or roll back to savepoint) |
| `Model::transaction(Closure)` | Run a callable within a transaction (auto-commit/rollback) |

---

## 3. Nested Transactions (Savepoints)

The framework supports **nested transactions** via database savepoints. The first call to `beginTransaction()` starts a real database transaction. Subsequent nested calls create savepoints (`SAVEPOINT trans_1`, `SAVEPOINT trans_2`, etc.).

### How Savepoints Work

```php
Model::beginTransaction();

// First operation
$order->status = 'pending';
$order->save();

// Create a savepoint (nested transaction)
Model::beginTransaction();

// Second operation (might fail)
try {
    $inventory->reduceStock($order->items);
    Model::commit(); // Releases savepoint
} catch (\Throwable $e) {
    Model::rollBack(); // Rolls back to savepoint, NOT to the beginning
    // $order is still saved as "pending"
}

// More operations...
$payment->process($order);
$payment->save();

Model::commit(); // Commits everything
```

### Nested Transaction Rules

| Level | `beginTransaction()` | `commit()` | `rollBack()` |
|-------|---------------------|------------|--------------|
| Outer (level 1) | Starts real DB transaction | Commits all changes | Rolls back everything |
| Nested (level 2+) | Creates a savepoint | Releases the savepoint | Rolls back to the savepoint |

> [!TIP]
> The outer `rollBack()` always undoes everything, including all nested savepoints. Only inner `rollBack()` targets the savepoint.

## 4. When to Use Transactions

### Always Use Transactions For:

**Financial operations:**
```php
Model::transaction(function () use ($invoice, $payment) {
    $invoice->markAsPaid();
    $payment->record($invoice->total);
    $account->addBalance($invoice->total);
});
```

**Multi-step operations:**
```php
Model::transaction(function () use ($order) {
    $order->status = 'confirmed';
    $order->save();

    foreach ($order->items as $item) {
        $item->product->decrementStock($item->quantity);
    }

    $order->notifications()->create([
        'type' => 'order_confirmed',
        'message' => "Order #{$order->id} confirmed",
    ]);
});
```

**Bulk updates:**
```php
Model::transaction(function () use ($newPrice) {
    Product::where('categoryId', 5)->update(['price' => $newPrice]);
    Product::where('categoryId', 6)->update(['price' => $newPrice * 0.9]);
    Product::where('categoryId', 7)->update(['price' => $newPrice * 0.8]);
});
```

### Don't Need Transactions:

- Single read or write operations
- Operations that don't depend on each other

## 5. How Write Connections Work During Transactions

When a transaction is active, all database operations automatically use the **write** connection. This ensures that all queries within the transaction see the uncommitted changes:

```php
Model::beginTransaction();

// Even though these are reads, they use the 'write' connection
// so they can see uncommitted changes from within the transaction
$sender = User::find($fromId);
$receiver = User::find($toId);

// Writes also go through the same write connection
$sender->balance -= $amount;
$sender->save();

Model::commit();
```

## 6. Complete Working Example

```php
class TransferService
{
    public function transferFunds(int $fromId, int $toId, float $amount): void
    {
        Model::beginTransaction();

        try {
            $sender = User::find($fromId);
            $receiver = User::find($toId);

            if (!$sender || !$receiver) {
                throw new \RuntimeException('User not found');
            }

            if ($sender->balance < $amount) {
                throw new \RuntimeException('Insufficient funds');
            }

            $sender->balance -= $amount;
            $sender->save();

            $receiver->balance += $amount;
            $receiver->save();

            TransactionLog::create([
                'fromId' => $fromId,
                'toId' => $toId,
                'amount' => $amount,
                'type' => 'transfer',
            ]);

            Model::commit();
        } catch (\Throwable $e) {
            Model::rollBack();
            throw $e;
        }
    }
}
```

Using the helper:

```php
class TransferService
{
    public function transferFunds(int $fromId, int $toId, float $amount): void
    {
        Model::transaction(function () use ($fromId, $toId, $amount) {

            $receiver = User::find($toId);

            if (!$sender || !$receiver) {
                throw new \RuntimeException('User not found');
            }

            if ($sender->balance < $amount) {
                throw new \RuntimeException('Insufficient funds');
            }

            $sender->balance -= $amount;
            $sender->save();

            $receiver->balance += $amount;
            $receiver->save();

            TransactionLog::create([
                'fromId' => $fromId,
                'toId' => $toId,
                'amount' => $amount,
                'type' => 'transfer',
            ]);
        });
    }
}
```

## 7. Transaction Best Practices

### Keep Transactions Short

Long-running transactions hold database locks and can cause contention:

```php
// BAD: Sending emails inside the transaction
Model::transaction(function () use ($user) {
    $user->save();
    $this->sendEmail($user->email); // Slow! Holds locks!
});

// GOOD: Send email outside the transaction
Model::transaction(function () use ($user) {
    $user->save();
});
$this->sendEmail($user->email);
```

### Use the Helper When Possible

The `transaction()` helper is preferred because it handles begin, commit, and rollback automatically:

```php
// Preferred: Clean, safe, automatic
Model::transaction(function () {
    // your operations
});
```

### Always Roll Back on Failure

Uncommitted transactions can leave locks hanging and cause deadlocks:

```php
// BAD: No rollback on error
Model::beginTransaction();
$user->save(); // Throws exception
// Transaction stays open! Locks held!

// GOOD: Always roll back
try {
    Model::beginTransaction();
    $user->save();
    Model::commit();
} catch (\Throwable $e) {
    Model::rollBack();
    throw $e;
}
```

> [!WARNING]
> Not all database engines support transactions equally. MyISAM tables in MySQL do not support transactions. Use InnoDB (or equivalent) for transactional tables.
