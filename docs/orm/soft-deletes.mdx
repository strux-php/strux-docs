# Soft Deletes: Safe Deletion

When a user deletes a record, sometimes you need to keep it — not because you want to show it, but because you might need to recover it later. This is where **Soft Deletes** come in.

Instead of actually removing the row from the database, soft deletes mark it as "deleted" by setting a timestamp. The record still exists in the database, but normal queries ignore it.

> [!INFO]
> Soft Deletes are implemented via the `HasSoftDeletes` trait, which your model must explicitly use.

---

## 1. Enabling Soft Deletes

### Step 1: Add the Trait and Attribute

```php
use Strux\Component\Database\ORM\Model;
use Strux\Component\Database\ORM\Behavior\HasSoftDeletes;
use Strux\Component\Database\Schema\Attributes\SoftDelete;

#[SoftDelete]
class Post extends Model
{
    use HasSoftDeletes;
}
```

### Step 2: Add the Deleted At Column to Your Table

```php
// In your Entity class (or migration)
#[Column(type: ColumnType::DATETIME, nullable: true)]
protected ?DateTimeImmutable $deletedAt = null;
```

> [!NOTE]
> The column name defaults to `deleted_at`. If you want to use a different name, override it in the `#[SoftDelete]` attribute.

---

## 2. How Soft Deletes Work

Once soft deletes are enabled, calling `delete()` on a model does not remove the row — it sets the `deleted_at` field to the current timestamp:

```php
$post = Post::find(1);

// Normal: returns the post
var_dump($post->deletedAt); // null

// Delete the post (soft delete)
$post->delete();

// Now: deletedAt is set
var_dump($post->deletedAt); // 2026-06-24 15:30:00

// The record is still in the database, but...
// Post::all() will NOT include it
```

Behind the scenes, this happens:

```
Before delete:
| id | title           | deleted_at |
|----|-----------------|------------|
| 1  | How to Code     | NULL       |

After delete (soft):
| id | title           | deleted_at          |
|----|-----------------|---------------------|
| 1  | How to Code     | 2026-06-24 15:30:00 |
```

> [!CAUTION]
> The row is never removed. If you have unique indexes on certain columns, a soft-deleted record can prevent creating a new record with the same value.

---

## 3. Querying with Soft Deletes

### Normal Queries (Exclude Soft-Deleted Records)

By default, `HasSoftDeletes` adds a global scope that filters out soft-deleted records:

```php
// Only returns non-deleted posts
$posts = Post::all();
// SQL: SELECT * FROM posts WHERE deleted_at IS NULL

$post = Post::find(1);
// SQL: SELECT * FROM posts WHERE id = 1 AND deleted_at IS NULL
// Returns null if post 1 was soft-deleted
```

### Including Soft-Deleted Records

Use `withTrashed()` to include soft-deleted records in the query:

```php
// Returns ALL posts, including soft-deleted ones
$allPosts = Post::withTrashed()->get();
// SQL: SELECT * FROM posts

// Find a specific post even if it's deleted
$post = Post::withTrashed()->find(1);
```

### Only Soft-Deleted Records

Use `onlyTrashed()` to get ONLY the soft-deleted records:

```php
// Returns only deleted posts
$deletedPosts = Post::onlyTrashed()->get();
// SQL: SELECT * FROM posts WHERE deleted_at IS NOT NULL
```

---

## 4. Restoring Soft-Deleted Records

Use the `restore()` method to bring a soft-deleted record back:

```php
$post = Post::withTrashed()->find(1);

if ($post->trashed()) {
    // This post was soft-deleted
    $post->restore();
    // Now deleted_at is set to null again
}
```

After restore:
```
| id | title           | deleted_at |
|----|-----------------|------------|
| 1  | How to Code     | NULL       |
```

---

## 5. Force Deletion (Permanent)

If you truly want to remove a record, use `forceDelete()`:

```php
$post = Post::withTrashed()->find(1);

// Permanently remove from database
$post->forceDelete();
// SQL: DELETE FROM posts WHERE id = 1

// The record is gone forever!
```

> [!WARNING]
> `forceDelete()` removes the record from the database permanently. There is no way to recover it afterward.

---

## 6. Checking Deletion Status

The trait provides several helper methods to check the deletion status:

| Method | Returns | Description |
|--------|---------|-------------|
| `trashed()` | `bool` | Whether this record is soft-deleted |
| `trashedById(int $id)` | `bool` | Whether a record with the given ID is soft-deleted (static) |

```php
$post = Post::withTrashed()->find(1);

if ($post->trashed()) {
    echo 'This post is in the trash.';
}

// Static check
if (Post::trashedById(1)) {
    echo 'Post 1 has been soft-deleted.';
}
```

---

## 7. Complete Method Reference

### Model Instance Methods

| Method | Description |
|--------|-------------|
| `delete()` | Soft-delete (sets `deleted_at` to now) |
| `forceDelete()` | Permanently remove from database |
| `restore()` | Restore a soft-deleted record |
| `trashed()` | Check if this record is soft-deleted |

### Query Builder Methods

| Method | Description |
|--------|-------------|
| `withTrashed()` | Include soft-deleted records in results |
| `onlyTrashed()` | Return only soft-deleted records |

### Static Methods

| Method | Description |
|--------|-------------|
| `trashedById(int $id)` | Check if a record by ID is soft-deleted |

---

## 8. Complete Example

```php
#[SoftDelete]
class User extends Model
{
    use HasSoftDeletes;

    // ... columns including 'deletedAt'
}

// ----- Normal Operations -----
$user = User::find(1); // Only if not deleted

// ----- Delete -----
$user->delete(); // Soft delete (sets deleted_at)

// ----- Querying -----
User::all(); // Only non-deleted
User::withTrashed()->get(); // All users
User::onlyTrashed()->get(); // Only deleted users

// ----- Restore -----
$deletedUser = User::withTrashed()->find(1);
$deletedUser->restore();

// ----- Force Delete -----
$user->forceDelete(); // Permanently gone

// ----- Check Status -----
$user->trashed(); // true or false
User::trashedById(1); // true or false
```

## 9. Best Practices

### When to Use Soft Deletes

✅ **DO use soft deletes when:**
- Users might accidentally delete data
- You need an "undo" or "trash" feature
- Compliance requires keeping records (GDPR "right to erasure" is an exception)
- You want to keep a historical trail

❌ **DON'T use soft deletes when:**
- The data is transient (logs, temporary data)
- Storage is a concern (soft-deleted records still take space)
- You have unique indexes that would conflict with soft-deleted rows

### Handling Unique Constraints

Soft-deleted records can cause issues with unique columns:

```php
#[Table(uniqueConstraints: [
    new UniqueConstraint(columns: ['email'])
])]
class User extends Model
{
    use HasSoftDeletes;
}

// User with email "test@example.com" is deleted
// You cannot create a new user with the same email
// because the deleted row still has it!
```

**Solutions:**
1. Use a composite unique index that includes `deleted_at`
2. Check for conflicts manually before creating
3. Allow the conflict and handle it at the application level

> [!TIP]
> When querying regularly, the global scope automatically excludes deleted records. You usually don't need to think about soft deletes until you specifically want to find or restore deleted records.
