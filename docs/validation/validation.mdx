# Validation in Strux

Welcome to the world of Strux Validation! If you're building an application, you're essentially building a gateway for data. People will send you names, emails, prices, and passwords. But here's a secret every senior developer knows: **Users (and bots) send bad data.**

Maybe they forgot to type their name. Maybe they typed "not-an-email" in the email box. Or maybe a hacker is trying to send a piece of code into your database.

Validation is your application's **Bouncer**. It stands at the door, checks every piece of data trying to get in, and says: *"Sorry, you can't come in until you look like a real email address."*

In Strux, we've designed this bouncer to be incredibly powerful, completely customizable, and—most importantly—easy to understand.

---

## 1. The Core Concept

At its simplest level, validation in Strux requires three things:
1.  **The Data:** Usually an array of information from a form (like `$_POST`).
2.  **The Rules:** A list of requirements for that data.
3.  **The Validator:** The engine that compares the data to the rules.

### A Basic Example

Imagine you have a simple signup form. Here is how you'd check the data manually:

```php
use Strux\Component\Validation\Validator;

$validator = new Validator();

// 1. The user's input
$userInput = [
    'username' => 'jd',
    'email'    => 'invalid-email'
];

// 2. Our rules
$rules = [
    'username' => ['required', 'minLength[3]'], // Note the new bracket syntax!
    'email'    => ['required', 'email']
];

// 3. Run the check
if ($validator->validate($userInput, $rules)) {
    echo "Everything looks great!";
} else {
    // Oh no, something is wrong. Let's see what!
    $errors = $validator->getErrors();
    print_r($errors);
}
```

In this case, the bouncer would reject the data because the username is too short and the email isn't real. The `$errors` array would automatically contain friendly messages explaining exactly what happened.

## 2. The Unique Bracket Syntax

One of the most distinct features of the Strux framework is our **Bracket Syntax** for rules. While other frameworks use colons (`min:3`), Strux uses brackets (`min[3]`). 

Why? Because it's more expressive and uniquely identifies your application as a Strux-powered project. It feels more like calling a function with parameters.

### How to use it:
- **`minLength[10]`**: "The text must be at least 10 letters long."
- **`range[1, 100]`**: "The number must be between 1 and 100."
- **`in[admin, editor, viewer]`**: "The value must be one of these three options."
- **`unique[users, email]`**: "Go check the 'users' table and make sure no one else has this email."

## 3. Conversational Error Messages

Default error messages can be boring. *"The field is required"* isn't very friendly. 

In Strux, you can turn your rules into a conversation using **Associative Arrays**. Instead of just giving the Validator a list of rules, you give it a map of **Rule => Message**.

```php
$rules = [
    'full_name' => [
        'required'     => 'We really need your name so we can say hi!',
        'alpha'        => 'Names can only have letters. No robot names allowed!',
        'range[3, 50]' => 'Your name must be between 3 and 50 characters.'
    ]
];
```

Now, if the user leaves the name blank, they'll see your custom "friendly" message instead of a generic framework error. This is a great way to give your application a unique personality!

## 4. Reusable Rules (The Rule Class)

Sometimes you have a rule that is very specific to your business. Maybe every "Coupon Code" in your store must start with `SAVE-`. 

Instead of writing complex code every time you use a coupon, you can create a **Reusable Rule Class**.

### Step 1: Create the Class
Create a new file in `src/Validation/Rules/Coupon.php`:

```php
namespace App\Validation\Rules;

use Strux\Component\Validation\Rules\RulesInterface;

class Coupon implements RulesInterface
{
    /**
     * This is the engine of your rule.
     * @param mixed $value The data the user typed.
     * @param mixed $data The entire form data (if you need to compare fields).
     * @return string|null Return an error message if invalid, or NULL if valid.
     */
    public function validate($value, $data = null): ?string
    {
        if (empty($value)) return null; // Let 'required' handle empty fields

        if (!str_starts_with((string)$value, 'SAVE-')) {
            return "All coupons must start with 'SAVE-'.";
        }

        return null; // Passed!
    }
}
```

### Step 2: Userland Overrides (The Magic)

This is the most powerful part of the Strux architecture. When you use a rule name like `'coupon'`, Strux searches for the code in this order:

1.  **Your Application:** `App\Validation\Rules\Coupon`
2.  **The Framework:** `Strux\Component\Validation\Rules\Coupon`

This means you can use your custom rule as a simple string!
```php
$rules = [
    'promo_code' => ['required', 'coupon'] // Strux finds your class automatically!
```

**Note for Masters:** This also means you can **override the framework**. If you don't like the built-in `Email` rule, just create `App\Validation\Rules\Email`. Strux will instantly start using *your* version instead of its own!

## 5. Built-in Rules Reference

Strux comes pre-loaded with everything you need to build a professional application.

| Rule | Syntax | What it checks |
| :--- | :--- | :--- |
| **Required** | `'required'` | Ensures the field isn't empty or null. |
| **Email** | `'email'` | Ensures the data is a valid email format. |
| **Numeric** | `'numeric'` | Ensures the data is a number (integer or decimal). |
| **Integer** | `'integer'` | Ensures the data is a whole number. |
| **Alpha** | `'alpha'` | Letters only (A-Z). |
| **Alphanumeric** | `'alphanumeric'` | Letters and numbers only. |
| **Slug** | `'slug'` | URL-friendly text (lowercase, numbers, and hyphens). |
| **MinLength** | `'minLength[n]'` | Text must have at least `n` characters. |
| **MaxLength** | `'maxLength[n]'` | Text cannot exceed `n` characters. |
| **Range** | `'range[min, max]'` | Number must be between `min` and `max`. |
| **In** | `'in[a, b, c]'` | Value must be one of the listed options. |
| **Same** | `'same[field]'` | Value must match another field (perfect for "Confirm Password"). |
| **Unique** | `'unique[table, col]'` | Ensures the value doesn't already exist in that database table. |

## 6. Validation in the ORM (Auto-Validation)

While you can run validation manually, the most common way to use it is directly inside your **Models**.

By adding the `#[Validate]` attribute to your model properties, Strux will automatically run the validation every single time you try to `save()`.

```php
use Strux\Component\ORM\Attributes\Validate;

class Product extends Model
{
    #[Column]
    #[Validate([
        'required' => 'Please name your product.',
        'range[5, 100]' => 'The price must be between $5 and $100.'
    ])]
    public float $price;
}

$product = new Product();
$product->price = 2.00;

if (!$product->save()) {
    // The save was blocked because the price was too low!
    $errors = $product->getErrors();
}
```

---

## Summary Best Practices

-   **Always Validate:** Never assume the data from a user is correct.
-   **Be Descriptive:** Use associative arrays to provide helpful error messages.
-   **Keep it Reusable:** If you use a custom regex more than twice, turn it into a Rule class in `App\Validation\Rules`.
-   **Double Defense:** Use validation in your **Forms** for immediate feedback, and in your **Models** for permanent database integrity.

Happy validating!
